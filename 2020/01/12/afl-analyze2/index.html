AFL源码分析二

源码分析

afl-fuzz.c
afl-fuzz.c是fuzz的核心部分，这里我们关注几个方面

  1. forkserver
  2. 共享内存
  3. 变异策略

forkserver

这是一种不使用execve()，而使用fork来提高效率的方法
afl的作者的博客中对这个问题做了一定的解释
https://lcamtuf.blogspot.com/2014/10/fuzzing-binaries-without-execve.html

首先来看看fork和exec类方法的区别

fork()
1、子进程复制父进程的所有进程内存到其内存地址空间中。父、子进程的“数据段”，“堆栈段”和“代码段”完全相同，即子进程中的每一个字节都和父进程一样
2、子进程拥有自己的进程ID
···
exec()
1、进程调用exec()后，将在同一块进程内存里用一个新程序来代替调用exec()的那个进程，新程序代替当前进程映像，当前进程的“数据段”， “堆栈段”和“代码段”被新程序改写。
2、新程序会保持调用exec()进程的ID不变。
···

可以理解为，fork是替身，而exec是换身

总结来说，exec类的方法因为使用的是新的程序来代替调用它的进程，所以对于fuzz这种需要通过大量的输入来执行程序，并且需要高效的找到程序问题的程序，是吃不消的。
所以afl的作者就使用fork来执行程序，其优点： 因为当前在fuzz的程序已经装载好了，所以可以利用fork的写时复制的特点，这样就可以很快的产生一个新的子进程，节省了载入目标文件和库、解析符号地址等重复性工作

创建forkserver

调用过程：
EXP_ST void init_forkserver -> static u8 calibrate_case -> static void perform_dry_run

perform_dry_run：对所有输入文件运行目标程序，保证运行是预期的。这个函数只会在开始调用一次

父进程Fuzzer

父进程Fuzzer通过fork()产生子进程forkserver，并使用st_pipe和ctl_pipe与子进程进行通信

父进程Fuzzer打开与forkserver的管道（分别是ctl_pipe的输入管道和st_pipe的输出管道），并等待forkserver的建立
当forkserver建立成功后，forkserver就会发送一个4字节的信息来通知父进程Fuzzer

子进程forkserver

对于子进程forkserver来说，会有一系列的设置【比较重要的是FORKSRV_FD为命令管道的输入，FORKSRV_FD+1为状态管道的输出】，最后使用execv去执行目标程序

execv函数
执行之后永远不会返回(除非出错)，且运行程序会替换当前进程，除了pid不变

设置好管道后，就可以进行状态的通知和接收了
回想上一篇文章（afl-as的介绍），我们插入了一段很长的汇编代码（afl-as.h）在每个分支中

这里主要做的是forkserver向父进程Fuzzer发送hello（FORKSRV_FD + 1），表示forkserver已经建立完毕，可以开始fork了

forkserver的fork

接下来看看forkserver是如何产生子进程来对目标程序进行fuzz的
这里选择main_payload_32进行分析，能更加清楚的了解其具体流程

下面参考了 http://rk700.github.io/2017/12/28/afl-internals/ 的内容

接着forkserver会一直监听命令管道（FORKSRV_FD），等待父进程Fuzzer的fork指令

当接收到了fuzzer的指令，则马上进行fork操作
这里fork了之后分为了又分为了父进程和子进程，父进程的pid（存储在%eax中）大于零，子进程的pid等于零。如果fork错误，则跳转到__afl_die，中断fuzz

流程图：

先看父进程forkserver

父进程forkserver所做的是将子进程的pid通过状态通道（FORKSRV_FD + 1）发送给forkserver的父进程Fuzzer
接着forkserver等待子进程targetApp执行完毕（waitpid）返回，forkserver再通过状态通道将结束状态发给Fuzzer，最后无条件跳转返回到__afl_fork_wait_loop中，重新进入等待状态

子进程targetApp

子进程targetApp在这里会关闭不再需要的管道，并继续执行TargetApp

流程图：

run_target()

在forkserver启动后，Fuzzer就会通过调用run_target来对目标程序进行fuzz

通过状态管道，父进程Fuzzer能够得到子进程的pid

特殊模式

在这个dumb_mode模式下，不使用forkserver，而直接使用execve进行fuzz

随后，fuzzer再次读取状态管道，获取子进程退出状态

通过状态，判断子进程结束的原因，例如FAULT_CRASH（崩溃），FAULT_ERROR（错误），FAULT_NONE（正常退出），并进行记录

总结

afl-forkserver的原理比较简单，使用fork来代替execve的方法
因此在fuzz的过程中，就只需要初始化init_forkserver的时候执行一次目标程序的准备，之后再对目标程序的测试，就不需要在重新做一遍目标程序的准备了（因为fork），这样就能够大幅的提高fuzz的效率
还有一点就是，在afl-fuzz对forkserver进行初始化的时候，会fork出一个新的进程，这个进程就是forkserver进程，这个进程和主进程通过管道进行通信。而真正的 forkserver 其实位于进程内，是在插桩的时候放入的
