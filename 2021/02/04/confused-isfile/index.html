本篇文章介绍了我踩一个小坑（isfile函数）的全部过程，实际原理并不复杂

起因

在php代码审计过程中，遇到了一个”能够”文件包含的点：

该框架使用的是thinkphp，并且当前的路由为/home/admin/xxx/fetch，同时template可控。那么此时如果有一个上传好的webshell，我们就可以通过这个路由的文件包含漏洞来利用这个webshell

如果需要进行文件包含，那么就不能进入if (!is_file($template)) { 的逻辑中，但每次thinkphp都给我返回了“模板不存在”的错误

起初我一直以为是路径写错了，所以在一直调整$template的值，同时也用file_get_content($template)去读取webshell的内容，结果是正确的，说明路径没错

那么问题来了，难道问题出现在is_file身上吗？

结果

最后在stackoverflow的一个回答中找到了答案
这个回答者也遇到了这个问题，并且他给了一个关于linux的手册，我则在这里找到了linux对于路径的]判断方式] (https://linux.die.net/man/7/path_resolution)
linux path resolution 有以下几个步骤：
Step 1: start of the resolution process
Step 2: walk along the path
Step 3: find the final entry

在Step 2中，linux会一层一层的去通过分割’/‘来解析字符串（路径），如果某一层的路径不存在，那么就会抛出ENOENT错误，并且终止解析的过程

Step 2: walk along the path
Set the current lookup directory to the starting lookup directory. Now, for each nonfinal component of the pathname, where a component is a substring delimited by aq/aq characters, this component is looked up in the current lookup directory.

If the process does not have search permission on the current lookup directory, an EACCES error is returned (“Permission denied”).

If the component is not found, an ENOENT error is returned (“No such file or directory”).
…

因此这也是为什么is_file('./existfile.txt')返回true而is_file('./NOTEXISTPATH/../existfile.txt')返回false的原因

is_file

可以看到，php中is_file是由c语言中的S_ISREG来进行判断的，而S_ISREG是一个宏，其定义如下

S_ISREG(m)中传入的m是stat的结构体中的st_mode字段，那么向上，找stat进行初始化的源码:

函数stat调用了xstat 函数，而__xstat 函数定义在glibc/sysdeps/unix/sysv/linux/i386/xstat.c文件中。

最后定位到了INTERNAL_SYSCALL (stat64, err, 2, name, &buf64);

file_exists

而同样file_exists最终也是的调用了这个INTERNAL_SYSCALL，只不过路径不太一样

file_exists是VCWD_ACCESS来进行判断的，通过这里，可以找到VCWD_ACCESS()对应于标准C库中的access()

st_mode

S_IFMT是一个掩码（八进制表示），可以用来过滤出前四位表示的文件类型

而控制st_mode的值的属性有 文件类型，文件id以及文件的权限
