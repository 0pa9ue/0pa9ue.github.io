不正常的路径

如果要是传入is_file的是一个”不正常“的路径（./NOTEXIST/../EXISTFILE），那么is_file会返回false，这里直接通过调试来寻找原因

php_stat首先对传入的type做一系列的判断（这里的type实际上是文件操作函数的一个编号，php将文件操作函数进行编号，然后统一调用php_stat函数进行处理）

因为当前的type = FS_IS_FILE(0xc)，不属于IS_ACCESS_CHECK，所以来到了894行的php_stream_stat_path_ex，而php_stream_stat_path_ex是一个宏，实际上调用的是_php_stream_stat_path函数

跟进_php_stream_stat_path，发现在第1935行有一个动态调用，通过动态调试可以看出此刻调用的是php_plain_files_url_stater函数

来到php_plain_files_url_stater函数，此时的flags = 0x2 ,PHP_STREAM_URL_STAT_LINK = 1，所以调用的是VCWD_STAT，也就是系统函数stat进行判断

因为此时php_stream_stat_path_ex的返回值为-1，所以进入了if循环然后RETURN_FALSE，最终导致is_file(./NOTEXIST/../EXISTFILE)的返回值为false

st_mode

那么最后，这两个php的函数实际上是linux的系统调用函数，而存储文件状态和讯息的stat结构体就是关键

而传入S_ISREG函数的st_mode参数实际上是stat结构体中的一个成员

S_IFMT是一个掩码（八进制表示），可以用来过滤出前四位表示的文件类型

控制st_mode的值的属性有 文件类型，文件id以及文件的权限

在上文所说到的is_file函数，因为(((m) & S_IFMT) == S_IFREG)的判断结果为false，所以导致is_file返回false

结论

找了很久资料，最后在stackoverflow的一个回答中找到了答案
这个回答者也遇到了这个问题，并且他给了一个关于linux的手册，我在这里找到了linux对于路径的[判断过程] (https://linux.die.net/man/7/path_resolution)
linux path resolution 有以下几个步骤：
Step 1: start of the resolution process
Step 2: walk along the path
Step 3: find the final entry

在Step 2中，linux会一层一层的去通过分割’/‘来解析字符串（路径），如果某一层的路径不存在，那么就会抛出ENOENT错误，并且终止解析的过程

Step 2: walk along the path
Set the current lookup directory to the starting lookup directory. Now, for each nonfinal component of the pathname, where a component is a substring delimited by aq/aq characters, this component is looked up in the current lookup directory.

If the process does not have search permission on the current lookup directory, an EACCES error is returned (“Permission denied”).

If the component is not found, an ENOENT error is returned (“No such file or directory”).
…

因此这也是为什么is_file('./existfile.txt')返回true，而is_file('./NOTEXISTPATH/../existfile.txt')返回false的原因了

更多

并且file_get_contents不止能够打开文件，还可以打开各种流（这也是为什么在上文中我将”文件“引起来），关键点还是在于_php_stream_open_wrapper_ex函数

PHP中的流是什么？

我们知道PHP中的fopen函数可以打开本地文件、URL等并返回一个句柄，fread、fwrite函数能对资源句柄进行读写，fclose用于关闭资源。PHP如何做到使用一致API对不同数据源进行操作？答案是PHP引入了“流”的概念，在底层对操作进行抽象，带来的好处是上层可用同一套API

那么，_php_stream_open_wrapper_ex工作主要有两点：

  1. 调用php_stream_locate_url_wrapper函数获取协议包装器(wrapper)
  2. 调用包装器打开资源并返回流对象

php_stream_locate_url_wrapper内部会对路径做了一系列的解析，判断使用的协议并寻找对应的wrapper来打开数据源（详细分析见博客）
